# 开发标准

本文件是本项目的 AI 代码生成与协作标准。AI 在生成、修改或重构代码前，必须先阅读并遵守本标准。

## 总体原则

1. 生成代码前必须先理解当前项目结构、已有代码风格、模块边界和已有协议定义。
2. 不允许只为了完成功能而孤立生成代码，必须同时考虑适配性、可扩展性、可维护性、可读性、健壮性和性能。
3. 对于会影响接口、协议、数据结构、模块边界或运行行为的改动，必须先给出实现方案。
4. 实现方案必须包含需求理解、影响范围、核心设计、错误处理、兼容性影响、性能考虑和验证方式。
5. 方案生成后必须先进行自我审查，指出可能不合理、不完整或存在风险的地方，并给出修正后的方案。
6. 在用户确认方案前，不得直接生成或修改业务代码。用户明确要求直接生成的简单文档、注释或配置除外。
7. 修改代码时应保持改动范围清晰，不进行与当前需求无关的重构。
8. 代码应优先复用项目已有模式，避免引入不必要的抽象或依赖。
9. 服务端代码必须严格按照分层设计编写，明确 repo、logic、service 各层职责。

## 分层设计规范

服务端代码必须遵守 repo、logic、service 三层职责边界，不允许跨层混写。

1. repo 层负责与数据库或外部持久化存储直接交互，封装数据查询、写入、更新、删除等能力。
2. logic 层负责业务逻辑处理，可以调用 repo 层封装好的能力，但不应直接操作数据库。
3. service 层负责封装 logic 层能力，对外提供接口服务，例如 gRPC、HTTP 或其他 RPC 接入。
4. service 层只做参数转换、基础校验、调用 logic、组装响应，不应承载复杂业务逻辑。
5. logic 层不应依赖具体传输协议，避免直接处理 gRPC context、HTTP request 等协议对象。
6. repo 层不应反向调用 logic 或 service，保持依赖方向单一：service -> logic -> repo。
7. 新增功能时，应优先按职责拆分文件和包，避免把数据库访问、业务规则和接口响应写在同一个函数中。
8. 必要时可以加入新的中间层，但必须服务于更优雅、高效地解决需求问题，不能为了增加层级而增加层级。

## Proto 文件规范

生成或修改 `.proto` 文件时必须添加简短注释。注释用于帮助快速理解协议含义，不应写成长篇说明。

### 基本要求

1. service 和 rpc 需要添加简短注释，说明用途即可。
2. message 需要添加简短注释，说明结构含义即可。
3. message 字段优先使用同行注释，格式为：`string status = 1; // 状态`。
4. 字段注释应尽量短，表达字段含义即可。
5. 对于令牌、验证码、状态、ID、时间戳等关键字段，可简短说明具体用途。
6. 字段命名必须使用 proto 推荐的 snake_case。
7. 字段编号一旦发布后不得随意修改或复用。
8. 删除字段时应使用 `reserved` 保留字段编号和字段名，避免兼容性风险。

### 注释示例

```proto
// LogicService 逻辑服服务
service LogicService {
  // Login 登录
  rpc Login(LoginRequest) returns (AuthResponse);
}

// LoginRequest 登录请求
message LoginRequest {
  string account = 1; // 邮箱或用户ID
  string password = 2; // 密码
}

// AuthResponse 认证响应
message AuthResponse {
  string status = 1; // 状态，true或false
  string content = 2; // 服务端信息
  string access_token = 3; // 访问令牌，用于身份校验
  string refresh_token = 4; // 刷新令牌
}
```

## Go 代码注释规范

生成 Go 代码时，必须为函数添加函数注释，并在函数内部关键流程添加必要注释。

### 函数注释

1. 所有导出函数、导出类型、导出方法必须添加符合 Go 习惯的注释。
2. 非导出但承担明确业务职责的函数也必须添加注释。
3. 注释应说明函数的业务目的，而不是重复代码字面含义。
4. 注释语言以中文为主，保持简洁准确。
5. 注释末尾不要添加句号。
6. 对重要变量、配置字段、全局变量、常量和结构体字段的声明行，应添加简短注释说明变量意义。

示例：

```go
// main 主函数
func main() {
    fmt.Println("hello")
}
```

更推荐在业务代码中写成：

```go
// validateEmail 校验邮箱格式。
func validateEmail(email string) bool {
    // 校验邮箱合规性
    return emailRegex.MatchString(email)
}
```

### 函数内部注释

1. 关键流程必须添加注释，例如参数校验、权限校验、状态转换、网络调用、数据库事务、物理模拟、协议转换、错误处理等。
2. 简单赋值、普通循环、显而易见的返回语句不需要机械注释。
3. 注释必须说明当前流程的业务目的。
4. 注释不能掩盖复杂逻辑，必要时应拆分函数提升可读性。

示例：

```go
// Register 处理用户注册流程。
func Register(req RegisterRequest) (AuthResponse, error) {
    // 校验邮箱合规性
    if !validateEmail(req.Email) {
        return AuthResponse{Status: "false", Content: "invalid email"}, nil
    }

    // 校验验证码是否正确且未过期
    if !verifyCode(req.Email, req.Code) {
        return AuthResponse{Status: "false", Content: "invalid verification code"}, nil
    }

    // 创建用户并生成令牌
    return createUserAndTokens(req.Email)
}
```

## 实现前方案要求

当需求涉及代码生成、业务逻辑、协议设计、数据库设计、网络通信、服务端架构或跨文件改动时，AI 必须先输出实现方案并等待用户确认。

实现方案至少包含：

1. 需求理解：简短复述要解决的问题。
2. 影响范围：列出预计修改或新增的目录与文件。
3. 设计方案：说明核心结构、接口、数据模型和调用流程。
4. 兼容性：说明是否影响已有协议、接口或数据。
5. 健壮性：说明错误处理、参数校验、异常分支和边界情况。
6. 性能考虑：说明是否存在高频调用、批处理、锁竞争、内存分配或网络开销。
7. 验证方式：说明如何通过测试、编译、生成代码或运行命令验证结果。

## 自我审查要求

方案提交给用户前，AI 必须先进行自我审查。

自我审查需要检查：

1. 是否遗漏已有代码风格或项目结构。
2. 是否存在过度设计或不必要依赖。
3. 是否存在协议字段含义不清、命名不清或兼容性风险。
4. 是否存在错误处理不足、空值处理不足或边界条件缺失。
5. 是否存在性能风险，例如频繁跨语言调用、重复分配、阻塞主循环或无界 goroutine。
6. 是否存在未来扩展困难的问题。
7. 是否有更简单、更清晰或更贴合项目现状的做法。

自我审查后，必须给出修正后的最终方案，并明确等待用户确认。

## 代码质量要求

1. 代码命名必须表达业务含义，避免含糊缩写。
2. 函数应保持单一职责，避免把校验、业务处理、持久化和网络响应混在一个大函数中。
3. 错误必须被显式处理，不允许静默忽略关键错误。
4. 对外接口返回值必须语义稳定，避免调用方依赖字符串猜测隐含状态。
5. 高频路径应减少不必要分配和重复计算。
6. 并发代码必须明确 goroutine 生命周期、退出条件和 channel 关闭策略。
7. 涉及 cgo 或 PhysX 交互时，应优先使用批量调用，避免每帧对每个对象进行大量跨语言调用。
8. 生成代码后应尽可能运行格式化、编译、单元测试或 proto 生成校验。

## 交付要求

1. 实现完成后必须说明修改了哪些文件。
2. 必须说明已执行的验证命令及结果。
3. 如果某些验证无法执行，必须说明原因。
4. 不得声称未验证的内容已经验证通过。
5. 如果实现过程中发现需求存在不合理处，必须先反馈并给出改进建议，再继续实现。
